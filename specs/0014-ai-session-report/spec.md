## 1. Context & Problem

A finished call leaves behind a transcript (spec 0009) and, since the bookmarks
feature (spec 0013, commit `f6eb14a`), a list of moments participants flagged
as important. But a raw multilingual transcript — `[HH:MM:SS] Speaker (lang):
text` lines in whatever languages were spoken — is a poor artifact to share or
act on. Someone still has to read it all and distill decisions, action items
and open questions, across languages.

This spec adds the first paid AI feature: a one-click post-meeting report,
generated by Groq from the stored transcript (bookmarks included), in the
requester's language of choice, billed against the existing credit balance
(spec 0005). It also lays the shared AI plumbing — transcript rendering,
chunking, map-reduce condensing, per-minute billing math — that the sentiment
(spec 0015) and follow-up-email ([0016](../0016-follow-up-email/spec.md))
features reuse.

## 2. Goals / Non-Goals

**Goals**
- One-click Markdown report — structured (fixed sections) or freeform prose —
  from any session the requester took part in, in a chosen language, with
  optional free-text guidelines.
- Duration-based pricing: `CREDITS_REPORT_BASE + CREDITS_REPORT_PER_MINUTE ×
  ⌈minutes⌉` (defaults $0.05 + $0.002/min), previewed client-side before the
  charge.
- **User-favorable billing policy:** advisory pre-check → generate → atomic
  deduct. A model failure costs nothing (502); a failure of OUR OWN
  infrastructure after generation never voids what the user paid for.
- Bookmarked moments surface as their own report section when present.
- Long calls work day one: a 2h transcript (~100k+ chars) is condensed
  map-reduce style before the final prompt.
- Reports persist; any participant reads the latest one free; regenerating
  appends history instead of overwriting.

**Non-Goals**
- No caching/dedup of generation: every Generate is a fresh model call and a
  fresh charge (deliberate — format/lang/guidelines change the output; contrast
  sentiment's one-cached-result-per-session contract in spec 0015).
- No guest access — the session screen and these endpoints are auth-only.
- No streaming generation, job queue, or email delivery (delivery is spec
  [0016](../0016-follow-up-email/spec.md)); the POST blocks up to ~a minute.
- No PDF/export of the report itself — a copy-Markdown button only (the
  transcript keeps its own exports per spec 0009).

## 3. Requirements

- **R1 — Generate a report.** As a logged-in participant of a session with
  transcript events, I want an AI-written report, so I don't distill the
  transcript by hand.
  - *Given* my session, *when* I POST `/api/sessions/{id}/report` with optional
    `format` (`structured` default | `freeform`), `lang`, `guidelines`, *then*
    I get 201 with the Markdown, the model used, the `cost`, and my new
    `balance`.
  - *Given* no `lang`, *then* the report language falls back to my own
    participant language for that session, then `en`.
  - *Given* invalid input (unknown format, guidelines > 2000 chars, lang not
    `[A-Za-z0-9-]{1,8}`), *then* 400; an event-less session, *then* 422;
    no token / stranger / unknown session, *then* 401/403/404.
- **R2 — Report shape.** *Given* `structured` format, *then* the prompt demands
  exactly these `##` sections: Executive Summary, Key Points, Decisions, Action
  Items, *(Bookmarked Highlights — only when the session has bookmarks, so the
  model can't invent one)*, Open Questions — bullets inside, `"None."` for
  empty sections, entire report in the requested language. `freeform` yields
  flowing prose sections at the model's discretion.
- **R3 — User-favorable billing.** As a paying user, I am never charged for
  output I didn't get, and never lose output I paid for to VoxTranslate's own
  errors.
  - *Given* a balance below the cost, *then* the advisory pre-check returns the
    standard 402 `insufficient_credits` body (feature `ai_report`) BEFORE any
    Groq call, balance untouched, no ledger row.
  - *Given* a Groq failure, *then* 502 "you were not charged" — no charge, no
    ledger row.
  - *Given* the balance dropped below cost between pre-check and the atomic
    `deduct_feature`, *then* 402 and the report is WITHHELD (delivering would
    make 402-then-regenerate a free path).
  - *Given* any *other* deduct error (DB hiccup) after generation, *then* the
    report is delivered FREE and the failure logged loudly.
  - *Given* the `session_reports` insert fails after a successful charge,
    *then* the 201 still carries the Markdown (the user paid for it); it just
    won't appear in GET later.
- **R4 — Long transcripts & model retirement.** *Given* a transcript over 48k
  chars, *then* it is condensed first (each ≤24k-char chunk summarized by the
  fallback model, preserving who-said-what, decisions, numbers, dates, names)
  and the final call runs on the condensed text. *Given* the primary model id
  is rejected (4xx — e.g. decommissioned), *then* one retry on the fallback
  model; timeouts/5xx do not retry on fallback (not the model's fault).
- **R5 — Persistence & read-back.** *Given* a stored report, *when* any
  participant GETs `/api/sessions/{id}/report`, *then* they get the newest row
  (free) with `cost` as a JSON *number* (not rust_decimal's string); *given*
  none yet, 404. Regenerating inserts a new row — history is kept.
- **R6 — Client UI.** As a user on the session detail screen, I get an AI card
  (slot `#ai-report-slot`) with format/language selects, a guidelines textarea,
  and a live cost preview from `/api/billing/ai-pricing` mirroring the server
  formula; Generate is disabled (with a need-X-have-Y tooltip) when the cached
  balance can't cover it — the server's atomic deduct stays the real gate. A
  stored report renders in a Markdown viewer with Copy and a Regenerate button
  that reopens the form prefilled from the last request. Localized via 11
  `aiReport*` keys × 8 UI languages; the card hides for guests and for
  event-less sessions (the server would 422 anyway).

## 4. Design & Architecture

- **Components / files:**
  - `server/src/ai/mod.rs` — shared AI plumbing: `transcript_to_text`
    (`[HH:MM:SS] Name (lang): text` lines relative to session start, `[chat]`
    marker, trailing `BOOKMARKS` block), `chunk_lines` (line-boundary splits),
    `condense_transcript` (map-reduce over 48k chars, 24k chunks),
    `billed_minutes` (ceil, floor 1).
  - `server/src/ai/report.rs` — `report_cost`, `report_prompt` (pure, for
    tests), `generate_report` (primary `GROQ_REPORT_MODEL`, max_tokens 2048,
    temp 0.3, 30s timeout, 3 retries on 429, one fallback-model retry on 4xx),
    `save_report` / `latest_report` persistence.
  - `server/src/api.rs` — `report_generate`, `report_latest`, shared
    `session_gate` (404/403) and `insufficient_credits` (402 body), `ai_pricing`.
  - `server/src/billing.rs` — `deduct_feature` (from AI-bundle foundations
    `41e3bee`): row lock → balance check → update → ledger row with the
    feature as `kind` plus `feature`/`session_id`/`metadata` detail.
  - `server/src/groq.rs` — `lang_name()` made `pub` for report prompts.
  - `client/src/scripts/report-md.ts` — dependency-free, escape-first Markdown
    renderer (model output is untrusted: transcript content could otherwise
    inject HTML) + `estimateReportCost` mirroring the server formula.
  - `client/src/scripts/report.ts` — form + viewer in `#ai-report-slot`;
    `api.ts` — `AiReport` types, `fetchAiPricing` (page-lifetime cache),
    `fetchLatestReport`, `generateReport` (402-aware); `session-screen.ts`
    calls `initReportSlot(ref)`; styles in `pages/index.astro`.
- **Data model:** `session_reports` (migration `005_features.sql`, shipped with
  the foundations): id, session_id (FK, cascade), user_id, guidelines, format
  CHECK `structured|freeform`, lang, markdown, model, cost DECIMAL(10,6),
  created_at; index `(session_id, created_at DESC)`. Multiple rows per session.
- **Protocol / API:**
  - `POST /api/sessions/{id}/report` `{format?, lang?, guidelines?}` → 201
    report row (+`balance` when charged) · 400 validation · 401/403/404 gates ·
    402 insufficient · 422 no transcript events · 502 model failure ·
    503 billing/transcripts not configured (guest-mode backend).
  - `GET /api/sessions/{id}/report` → 200 latest stored report · 404 none yet.
  - `GET /api/billing/ai-pricing` → `"report": {"base", "per_minute"}` for the
    client preview (env-configured user prices only; raw cost/markup never
    exposed).
- **Sequence (happy path):** validate format/guidelines → transcript flush
  (barrier so a report requested right after leaving sees the final events) →
  `session_gate` → resolve lang (param > participant lang > `en`) → export
  transcript (422 if empty) → compute cost from `duration_seconds` → advisory
  balance pre-check → condense if long → Groq generate (fallback dance) →
  atomic `deduct_feature` → insert `session_reports` row → 201.
- **Key decisions:**
  - *Advisory pre-check + atomic deduct-after-generation* → fail fast before
    burning an expensive Groq call, without TOCTOU holes: the deduct is the
    real gate. Alternative (charge first, refund on failure) rejected — refund
    failure modes are strictly worse than deliver-free ones.
  - *Asymmetric deduct-failure handling* (402 withholds, any other error
    delivers free) → the only deduct error a user can cause is insufficient
    funds; everything else is our infra, and "charge-or-lose paid AI output
    over our own error" is never acceptable. Spec 0016's email draft reuses
    this policy verbatim.
  - *Regenerate appends, GET returns latest* → cheap history, no cache
    staleness; the requester pays per generation, every participant reads free.
    Sentiment (spec 0015) chose the opposite (UNIQUE per session, cached)
    because its input can't be steered per-request.
  - *`cost` re-serialized as a JSON number* (`report_json`) — rust_decimal
    serializes Decimal as a string; every other money field we expose is a
    number.
  - *Guidelines passed as data with explicit precedence rules* ("apply them
    where they don't conflict with the rules above"), never interpolated into
    directives → bounded prompt-injection blast radius.
  - *Hand-rolled escape-first client renderer* → the input is model output in
    a known Markdown dialect; a parser dependency isn't worth the surface.
  - *Map step on the cheap fallback model, reduce on the report model* →
    condensing is mechanical; quality budget goes to the final report.

## 6. Testing & Verification

- **Unit (server, 6 ai-module tests):** `report_cost` base + ceiled minutes
  (0s and 60s both bill 1 minute; 61s bills 2); `report_prompt` structured
  sections + language + no Bookmarked-Highlights-without-bookmarks (R2),
  freeform + guidelines passthrough; `transcript_to_text` renders speech, chat
  marker and bookmarks with relative timestamps; `billed_minutes` rounds up,
  floors at 1, tolerates negatives; `chunk_lines` splits losslessly on line
  boundaries.
- **Integration (`tests/transcripts.rs::ai_report_validation_billing_and_persistence`,
  DB-gated):** 401/403/404 gates on both verbs, 404 before any report exists,
  validation 400s (bad format, 2001-char guidelines, `p?q` lang), 422 empty
  session, 402 pre-check with balance untouched, Groq failure (the test
  config's dummy key makes the real POST path die at generation) → 502
  containing "not charged" + balance untouched + zero `ai_report` ledger rows
  (R3), GET round-trip with `cost` as a JSON number (R5), regenerate appends —
  GET returns the newest of 2 kept rows.
- **Unit (client, `report-md.test.ts`, 7 tests):** renderer headings (## → h3,
  deeper → h4), bullet/numbered lists closed on blanks, paragraph joining,
  HTML escaping before any tag insertion, `*emphasis*` at line start not
  mistaken for a bullet; `estimateReportCost` parity with the server formula.
- **Gates:** the repo-standard suite — server `cargo llvm-cov` ≥85% lines,
  client vitest thresholds, `astro check`, Playwright e2e — stayed green; the
  card itself is auth+billing-gated, so it's exercised manually (the guest
  e2e backend 503s these endpoints by design).

## 7. Deployment & Operations

- **Env (server):** `GROQ_REPORT_MODEL` (default `llama-3.3-70b-versatile`),
  `GROQ_FALLBACK_MODEL` (default `llama-3.1-8b-instant`),
  `CREDITS_REPORT_BASE` (default `0.05`), `CREDITS_REPORT_PER_MINUTE` (default
  `0.002`). Rides the existing `GROQ_API_KEY`; requires the billing stack
  (spec 0005) and transcripts (spec 0009) — without them the endpoints 503.
- **Migration:** `005_features.sql` (`session_reports`) runs at startup;
  shipped with the AI-bundle foundations commit.
- **Pricing changes** are env-only (no code) and reflected immediately in the
  client preview via `/api/billing/ai-pricing`.
- Rollout: server via `railway up` from `server/`; client autodeploys on push.
  Watch for `tracing::error!` lines `"deduction failed AFTER generation —
  delivering free"` and `"report insert failed after charge"` — both are
  money-relevant and should stay at zero.

## 8. Risks / Open Items

- **No generation idempotency:** a retried/double POST charges twice. The
  client disables Generate in flight, but that's client-side only.
- **Synchronous generation** (up to ~1 min in one HTTP request): if the client
  drops after the deduct, the user is charged but never sees the 201 — the row
  IS persisted, so GET recovers it on the next visit. The one truly lossy
  corner is the insert-failed-after-charge path, where the delivered Markdown
  exists only in that one response.
- **Free-delivery path is unbounded in principle** (a sustained deduct-DB
  outage would hand out free reports); accepted — it requires our own DB to be
  half-broken, and the error logs are the alarm.
- Client cost preview derives duration from `started_at`/`ended_at` (or
  `Date.now()` for unfinished sessions) and may drift a minute from the
  server's `duration_seconds`; harmless — the server bills its own number.
- Commit message numbering: `c2bc646` says "spec 0014 PR7" and happens to
  match, but the AI-bundle commits reused 0011–0015 inconsistently; the README
  feature map is canonical (this doc backfills 0014).
